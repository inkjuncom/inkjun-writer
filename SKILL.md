这是一个用于公众号长文写作的精简技能。

## 适用范围

当用户有这些意图时使用本技能：

- 想讨论长文选题、角度、结构，最终得到提纲
- 想从提纲或素材起草完整长文
- 想修改、改写、重构已有长文

## 不适用范围

出现以下情况时，不进入本技能：

- 只要短内容（朋友圈、小红书、推文、口播短稿）
- 只要标题或摘要，不要正文
- 只做一句话润色、错别字修正、标点修正
- 只问概念解释，不需要文章产出

## 模式选择

- 信息不足、目标不清，或用户想先聊清楚再动笔：用讨论模式
- 已有提纲、素材、或已有全文要修改/改写/重构：用起草模式

如用户同时提出多个目标，默认串联：讨论 → 起草。

## 工作模式

### 1) 讨论模式

用于把"写什么、写给谁、写到什么程度"聊清楚，最终输出提纲。

向用户提问，优先补齐：

- 目标读者
- 文章目标（希望读者看完有什么变化）
- 已有素材（事实、案例、数据、观点）
- 必须保留的判断
- 文体倾向（观点、叙事、教程、混合）

信息足够后，输出提纲：

- 标题方向 3~5 个
- 导语切口 2~3 个
- 一级结构 3~6 段，每段的核心问题
- 结尾收束方式

默认推进线：`为什么值得看 -> 核心判断 -> 论证展开 -> 收束到行动或提醒`

### 2) 起草模式

用于从提纲或素材生成完整长文，也用于修改、改写、重构已有长文。

**执行顺序：逻辑重构 → 信息扩充 → 信息精炼 → 语法检查 → 读取 few-shots → 起草/改写**

#### 逻辑重构（必做，需用户确认）

在动笔前，先审视大纲的逻辑组织是否最优：

- 检查段落顺序：现有顺序是否是读者最容易接受的路径
- 检查论点层级：哪些是核心判断，哪些是支撑论据，层级是否清晰
- 检查推进逻辑：段落之间的因果、转折、递进关系是否成立
- 提出 1~2 个备选结构方案，说明各自的优劣

**输出重构后的大纲，等待用户确认后再继续。**

#### 信息扩充（必做）

用户确认大纲后，在大纲框架内补充信息，不编造：

- 为每个论点补充具体支撑：场景、案例、数据、细节
- 标出当前素材中已有的信息来源
- 标出信息不足、需要用户补充的位置（用「待补：xxx」标注）
- 不因信息不足而编造案例或数据

#### 信息精炼（必做）

先对用户提供的素材、提纲或原文做一轮去冗：

- 合并表达同一意思的句子，只保留最清晰的一条
- 删除重复出现的判断或结论（同一观点换个说法再说一遍的，删掉后者）
- 标记信息密度极低的段落，注明"待补"或直接删除

#### 语法检查（必做）

精炼完成后，将文本拆分为独立的句子列表，**逐句**检查语法结构，每句必须过一遍，不得跳过或批量处理。重点排查以下四类问题：

- **主语缺失**：句子缺少显式主语，读者无法直接判断动作主体。即使上下文能推断，只要主语没有写出，即判定为缺失。特别注意以下常见口语省略模式：

  - **句首时间词 + 动作**：「最近/后来/回头/回来后/之前/以前 + 动作」→ 缺「我/它」
  - **句首连词 + 动作**：「不过/然后/于是/结果/接着 + 动作」→ 缺「我/它」
  - **句首处所 + 动作**：「终端/官网/GitHub/浏览器 + 动作」→ 缺「在/从」或「我」
  - **句首动作**：「去/来/试/看/找/点/敲/按/编/写 + 动作」→ 缺「我」
  - **情态动词开头**：「可以/能/要 + 动作」→ 缺「它/这」
  - **评价性省略**：「够了/不方便/刚刚好/挺不错」→ 缺「这/这样/但这样」
  - **动态省略**：「每次/一跑/一合上 + 就…」→ 缺「它」或补出具体主语

  可接受的省略：同一段落内、紧接上句、且主语明确一致时的承前省略。跨句、跨段、或主语发生切换时必须补全。
- **宾语缺失**：及物动词后缺少宾语，读者无法判断动作目标（如「每次我让 Hermes 起草」）
- **主谓搭配不当**：主语与谓语动词不搭配（如「核心问题浮出水面」中「问题」不能「浮出」）
- **动宾搭配不当**：动词与宾语搭配过于抽象，意思不落实（如「让约束更收束」）

检查方式：
- 按顺序为每句编号，逐句输出检查结果（无问题的句子也要标注"✓"，证明已检查）
- 发现问题时，标出问题句，注明问题类型，给出修改建议后直接修改
- 全部句子检查完毕后，汇总问题数量，再继续下一步

#### 读取 few-shots

精炼完成后，读取 `references/rewrite-few-shots.md`。

`references/rewrite-few-shots.md` 为最高优先级规则源：

- 输出前必须完成分级判定（L0/L1/L2）与 Gate 1~3 检查
- 规则冲突时，按 few-shot 执行

few-shot 使用规则：

- 学"改写动作"，不借原句
- 优先模仿拆句、降模板化转折、把判断写实
- 禁止把 few-shot 原句拼入正文

#### 写作规则

- 默认中文
- 先保证观点推进，再处理文采
- 多给具体信息（场景、动作、例子、数据），少给空结论
- 避免连续使用同一转折骨架
- 段落长短自然变化
- 小标题必须有信息量
- 材料不足时标出待补信息，不编造案例

#### 输出前自检（详细模式）

正文输出前，必须做一轮完整的重新扫描自检，以验证所有修复已生效。**不得只报数字，必须展示检查过程。**

##### 第一步：全文 L0 重扫

将改写后的全文拆成句子列表，逐句扫描以下三类 L0 违规，每条标注 ✅ 或 ❌：

- **破折号补充说明**：扫描全文是否出现 `——`（替换为句号/冒号）
- **主语缺失**：逐句检查主语是否存在。特别注意以下模式：
  - 「最近/后来/回头/回来后/之前/不过/然后/于是/结果/接着 + 动作」→ 缺「我」
  - 「去/来/试/看/找/点/敲/按/编/写 + 动作」→ 缺「我」
  - 「终端/官网/GitHub/浏览器 + 动作」→ 缺「在/从」
  - 「可以/能/要 + 动作」→ 缺「它/这」
  - 「就够了/刚刚好/不方便/挺不错」→ 缺「这/这样」
  连「我」被省略的也必须补回。
- **「不是A而是B」式转折**：扫描「不是…而是…」「并非…而是…」骨架

输出格式示例：
```
📍 L0 重扫（共 N 句）
  S1: …… ✅
  S2: …… ✅
  S3: …… ❌ 主语缺失（「回来后按 Control C 停掉」→ 补「我」）
```

全部通过则标注「L0：0 条 ✅ 清零成功」。

##### 第二步：L1 + AI腔 扫描

扫描全文模板化衔接词（首先/其次/第一/第二/综上所述）和 AI 腔短语（一句话总结/值得注意的是/简而言之/需要指出的是），列出具体位置。

```
📍 L1 监控：0 条 ✅
📍 Gate 1（AI腔）：0 条 ✅
```

##### 第三步：Gate 门禁逐项论证

**Gate 1（无AI腔）**：列出是否发现任何 AI 腔短语。通过/不通过。

**Gate 2（有场景支撑）**：说明每个核心判断是否有具体事实/动作/场景支撑。举例说明 1~2 处。通过/不通过。

**Gate 3（段落节奏）**：列出各段落字数，确认长度自然变化（2-4句为主，不过度整齐）。通过/不通过。

##### 第四步：语法问题汇总

列出改写后重新扫描发现的语法问题（主语缺失 / 宾语缺失 / 搭配不当），应与语法检查步骤的问题数一致或更少（表明已修复）。

##### 汇总表格

```
L0：0 条 ✅
L1：0 条 ✅
AI腔：0 条 ✅
语法问题：0 条 ✅
Gate 1：通过 ✅
Gate 2：通过 ✅
Gate 3：通过 ✅
```

Gate 未通过时，不输出终稿，先继续改写。
